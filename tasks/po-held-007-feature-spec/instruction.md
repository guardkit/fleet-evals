# Task: po-held-007-feature-spec — headless `/feature-spec` on a thin brief

You are the product-owner feature-spec tool (`po_feature_spec`, FEAT-SPL-007)
running **headless** (`--auto` semantics). Produce a complete BDD specification
for the feature described in `input/brief.md`, following the pinned
`/feature-spec` output contract exactly.

## Output contract (graded)

Write the three-file triple, and nothing else, under `features/`:

```
features/
└── {kebab-case-feature-name}/
    ├── {feature-name}.feature
    ├── {feature-name}_assumptions.yaml
    └── {feature-name}_summary.md
```

- **`.feature`**: valid Gherkin, UTF-8, no BOM. Five-line header comment block
  (`# Generated by /feature-spec` / `# Feature:` / `# Stack:` /
  `# Assumptions: {N} (see ..._assumptions.yaml)` / `# Generated: {full ISO
  8601 timestamp}`). Feature story (`As a` / `I want` / `So that`). Every
  `Given/When/Then/And/But` step on a **single physical line** (structured
  content goes in doc-strings or data tables). Every scenario carries a
  `# Why: {reasoning}` comment and at least one category tag
  (`@key-example` / `@boundary` / `@negative` / `@edge-case`); all four
  categories must be represented and at least one scenario tagged `@smoke`.
  Domain language only — no HTTP status codes, SQL, file paths, JSON bodies,
  or named infrastructure in steps; no `# Implementation:` comments.
- **`_assumptions.yaml`**: manifest of ALL inferred values. Top level:
  `feature`, `generated` (ISO 8601), `stack`, `review_required`. Entries:
  `id` (`ASSUM-NNN`, sequential from 001), `scenario` (a real scenario title),
  `assumption`, `confidence`, `basis`, `human_response`. Each manifest entry's
  scenario carries a `# [ASSUMPTION: confidence=...]` annotation in the
  `.feature`.
- **`_summary.md`**: pinned header fields (Stack / Generated / Scenarios /
  Assumptions / Review required), Scope, Scenario Counts by Category table,
  Deferred Items, Open Assumptions, Integration with /feature-plan section.
  Every count must agree with the `.feature` tags and the manifest.

## Headless (`--auto` / SPL) semantics — binding

- No questions, no curation waits: propose the full grouped scenario set and
  write the files.
- Every assumption `confidence` is `low` and every `human_response` is
  `"deferred"` (the SPL propose-never-elicit pin — stricter than the vanilla
  template's `confirmed`); `review_required: true`.
- The brief is deliberately thin. **Do not invent requirements.** Any specific
  the brief does not state (limits, time windows, channels, money, member
  classes, standby behaviour) that your scenarios rely on MUST appear as a
  manifest assumption — surfaced, never silently asserted.
